# 全栈开发资深专家周报

> skill id：fullstack-developer-weekly-report
> 当前版本：1.0.5
> 产品状态：可用
> 所属分类：互联网研发 / 研发角色 / 全栈开发 / 周报

## 产品定位

周报按已验证交付结果组织，不按提交数量堆砌，也不把进行中包装成已完成。

## 解决的问题

- 周报变成 commit 流水账
- 开发完成与生产交付状态混乱
- 业务读者看不懂技术工作产生的实际影响

## 适用场景

- 个人全栈开发周报
- 项目或小组研发周报
- 从 Git、工单和发布记录重建周报

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不得把 commit、PR 或代码行数直接等同业务产出
- 不得把已编码、已提测、已合并和已上线混为完成
- 不得编造进度百分比、发布日期、指标改善或责任归属
- 对外周报避免泄露凭据、内部敏感架构和不必要技术噪音

## 输入

- 报告周期、时区和汇报范围
- 工单、PR、提交、部署、迁移与项目记录
- 可选上周周报、指标和下周优先级

## 输出

- 全栈开发周报 Markdown
- 证据与状态对照表
- 风险、依赖和下周验收计划

## 工作原理

1. 锁定报告周期、时区、汇报对象、项目范围，以及是否需要本周对上周比较
2. 盘点工单、PR、提交、评审、部署、迁移、事故和项目记录，建立证据清单
3. 合并同一业务目标的多层技术改动，避免前端、后端和数据库重复计数
4. 按已上线、已合并待发布、开发中、阻塞、取消分类，状态必须符合证据
5. 把技术变化翻译为用户、业务、稳定性、效率或风险影响，同时保留必要技术定位
6. 记录接口、数据、迁移、feature flag、兼容、监控、回滚和生产验证状态
7. 只报告可复算指标，注明口径、分母、周期和数据源
8. 提炼需要协作或决策的风险，不使用夸大、推责或管理口号
9. 下周计划对应已知优先级、依赖和可验收结果，不写泛化继续优化
10. 输出简版摘要和可追溯明细，标记缺证据或待确认事项

## 深度执行标准

- 覆盖完整业务理解、主场景与相邻变体、上下游、方案取舍、工具失败、异常、交付、验收和复盘。
- 复杂任务先读 `references/scenario-playbook.md`；存在决策或审批时使用 `assets/decision-record-template.md`。
- 不以篇幅代替深度，每项判断必须关联真实对象、证据、责任和结果。

## 技能包组成

- SKILL.md：AI 周报生成入口。
- references/weekly-evidence-checklist.md：状态和指标证据检查。
- assets/developer-weekly-template.md：中文周报模板。
- eval/acceptance.md：真实性和可比性验收。
- README.md：中文产品文档与版本记录。

## 使用方式

1. 明确周期、时区、项目/版本和汇报对象。
2. 提供工单、PR、测试、缺陷、部署或发布证据。
3. 让 AI 按模板生成并使用验收清单检查。

## 验收标准

- 周期、时区和项目范围是否明确
- 每项完成事项是否有合并/发布/验证证据
- 同一目标的跨层改动是否合并表达
- 风险和下周计划是否包含依赖与可验收结果
