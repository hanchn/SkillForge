# 全栈测试资深专家

> skill id：fullstack-quality-engineer
> 当前版本：1.0.6
> 产品状态：可用
> 所属分类：互联网研发 / 研发角色 / 全栈测试

## 产品定位

质量不是测试用例数量，而是高风险用户和系统行为都有可信证据、可定位失败和恢复路径。

## 解决的问题

- 每层测试都通过但完整用户旅程失败
- 测试数据、权限和环境差异造成假阳性
- 发布后才发现迁移、监控和回滚不可用

## 适用场景

- 验证跨前后端和数据库的完整功能
- 为重大版本做 release readiness
- 审查测试计划的跨层风险漏测

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不得用端到端用例替代所有低层精确验证
- 不得只测接口 200 或页面出现而忽略数据和副作用
- 不得在共享环境制造不可清理或敏感测试数据
- 不得因测试通过忽略观测、迁移和回滚不可验证

## 输入

- 需求、架构和变更范围
- 测试环境、数据、身份和依赖
- 发布、迁移、监控和事故背景

## 输出

- 全栈风险与覆盖矩阵
- 跨层测试结果和缺陷证据
- 发布 gate 与阻塞项
- 生产验证、回滚和数据修复方案

## 工作原理

1. 读取需求、架构、变更 diff、事故历史、数据流、发布计划和环境差异
2. 建立用户旅程到前端、API、服务、数据库、任务、事件和第三方的影响图
3. 按概率、影响、可检测性和爆炸半径建立风险清单
4. 为每项风险选择最低但可信的验证层和明确 oracle，避免层层复制同一场景
5. 设计身份、权限、租户、locale、时区、设备、网络、feature flag、数据和清理策略
6. 覆盖正常、边界、校验、权限、并发、重复、超时、重试、部分失败、回滚和兼容
7. 检查前端多语言、键盘、焦点、响应式和性能关键旅程
8. 检查 API 对象/字段授权、错误、幂等、资源限制、日志和敏感数据
9. 验证数据库迁移、backfill、约束、对账、旧版本共存和恢复
10. 定义发布 gate、生产 smoke、业务指标、技术告警、数据修复和回滚触发

## 深度执行标准

- 覆盖完整业务理解、主场景与相邻变体、上下游、方案取舍、工具失败、异常、交付、验收和复盘。
- 复杂任务先读 `references/scenario-playbook.md`；存在决策或审批时使用 `assets/decision-record-template.md`。
- 不以篇幅代替深度，每项判断必须关联真实对象、证据、责任和结果。

## 技能包组成

- SKILL.md：AI 执行入口。
- references/fullstack-quality-checklist.md：跨层专业检查表。
- assets/fullstack-test-matrix.md：端到端追溯模板。
- eval/acceptance.md：验收标准。
- README.md：中文角色产品文档和版本记录。

## 使用方式

1. 提供需求、仓库、环境和发布约束。
2. 让 AI 读取 SKILL.md 与检查表。
3. 使用矩阵保持跨层追溯并在交付前验收。

## 验收标准

- 每个高风险行为是否有测试层和可信 oracle
- 身份、数据、locale、时区和环境差异是否覆盖
- 迁移、异步、第三方和失败恢复是否验证
- 发布 gate、生产 smoke 和回滚触发是否可执行
