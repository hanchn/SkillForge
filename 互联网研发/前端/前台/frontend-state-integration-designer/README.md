# 前端状态与数据流设计师

> skill id：frontend-state-integration-designer
> 当前版本：1.0.0
> 产品状态：可用
> 所属分类：互联网研发 / 前端 / 前台 / 状态与数据

## 产品定位

每份状态只能有一个权威来源；能派生就不复制，服务端事实不伪装成本地真相。

## 解决的问题

- 同一数据在 props、store 和 cache 中漂移
- 快速切换或重复提交导致过期响应覆盖新状态
- 乐观更新失败后 UI 与服务端不一致

## 适用场景

- 复杂筛选、分页和 URL 同步
- 表单草稿、保存和并发编辑
- 接入查询缓存、mutation 与乐观更新

## 不适用范围与边界

- 不得把服务端数据无理由复制进全局 store
- 不得用 effect 链修补可计算状态
- 不得把 loading 当作唯一异步状态
- 不得乐观更新不可安全回滚或高风险操作

## 输入

- 功能需求与 UI 状态
- API 或事件契约
- 现有状态、缓存和路由方案

## 输出

- 状态清单和 ownership
- 查询/变更生命周期设计
- 竞态、错误和恢复矩阵
- 数据流图与实现建议

## 工作原理

1. 列出页面中所有状态及其所有者、生命周期、持久化和共享范围
2. 分类为 URL、服务端缓存、表单草稿、本地交互、全局客户端、环境或派生状态
3. 删除可派生副本并定义单一 source of truth 与同步边界
4. 为查询定义 key、参数、启用条件、新鲜度、缓存、失效、分页和取消
5. 为 mutation 定义验证、幂等、pending、重复提交、乐观更新、回滚、重新获取和错误呈现
6. 处理路由切换、过期响应、快速筛选、并发编辑、tab 多开和权限变化
7. 明确 loading、empty、partial、stale、error 和 retry 的 UI 语义
8. 定义敏感数据、持久化、日志和缓存清理边界
9. 绘制事件到状态再到渲染和副作用的数据流
10. 用竞态、失败、回退和恢复案例验证设计

## 使用方式

1. 提供目标、现有代码或生产证据与约束。
2. 让 AI 先读 SKILL.md 和指定 reference。
3. 按仓库实际命令验证，使用 asset 完成交付。

## 验收标准

- 每份状态是否有唯一 owner 和生命周期
- 服务端缓存与本地状态是否清楚分离
- 查询 key、失效和取消是否完整
- mutation 的重复、失败、回滚和并发是否可验证
