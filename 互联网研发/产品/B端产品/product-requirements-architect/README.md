# 产品需求架构师

> skill id：product-requirements-architect
> 当前版本：1.0.1
> 产品状态：可用
> 所属分类：互联网研发 / 产品 / B端产品

## 产品定位

将模糊产品想法或现有 PRD 转换为可评审、可开发、可测试、可度量的需求包。

## 解决的问题

- 问题陈述被写成功能方案，团队无法判断为什么做
- 状态、权限、异常、边界和非目标经常缺失
- 需求、验收标准和业务指标之间无法追溯

## 适用场景

- 把口头需求或工单整理为完整 PRD
- 审查现有需求文档是否具备开发与测试条件
- 设计审批、配置、权限或复杂状态工作流

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不替代产品负责人做未授权商业决策
- 不把假设包装成利益相关方已确认事实
- 不把内部技术实现当成产品需求，除非它是明确约束

## 输入

- 产品想法、需求简报或现有 PRD
- 可选访谈、工单、截图、约束和指标证据

## 输出

- 需求包或 PRD
- 需求—验收—指标追溯矩阵
- 风险、依赖和待决策清单

## 工作原理

1. 分离事实、假设、未知和排除项
2. 定义问题、角色、任务和系统边界
3. 建立主流程、状态、权限和异常
4. 编写原子化需求与可观察验收标准
5. 定义指标和 guardrail
6. 用追溯表检查孤立需求

## 技能包组成

- references/quality-gates.md：需求质量门槛
- assets/prd-template.md：完整 PRD 模板
- eval/acceptance.md：问题、需求与验收检查

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据
- INVOCATION.md：标准调用顺序与失败处理
- BASE_PROMPT.md：跨平台执行原则
- agents/openai.yaml：支持平台的界面元数据，如该文件存在
- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 product-requirements-architect/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 问题不是功能复述
- 需求可独立评审和测试
- 状态、权限与失败行为完整
- 每项重要需求能追溯到问题和指标

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
