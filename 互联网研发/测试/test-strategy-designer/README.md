# 测试策略设计师

> skill id：test-strategy-designer
> 当前版本：1.0.1
> 产品状态：可用
> 所属分类：互联网研发 / 测试 / 测试策略

## 产品定位

根据变更影响和风险，设计分层测试覆盖、发布门槛、回归范围和生产验证。

## 解决的问题

- 测试计划按用例数量堆砌，却没有覆盖最高风险
- 同一个场景在每一层重复，失败后仍难定位
- 环境差异、数据、身份、时钟、feature flag 和回滚信号被遗漏

## 适用场景

- 为新功能、API、迁移、事故修复或第三方集成制定测试策略
- 审查现有测试计划的高风险漏测
- 定义发布 gate、focused regression 和上线后验证

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不默认 staging 与 production 等价
- 不以测试数量衡量质量
- 不在每一层机械复制同一场景

## 输入

- 变更范围、需求或设计
- 可选架构、事故历史、环境和发布约束

## 输出

- 风险测试策略
- 风险—测试—oracle 覆盖矩阵
- 发布门槛、回归和生产验证方案

## 工作原理

1. 建立用户到服务、数据和依赖的影响图
2. 按概率、影响、可检测性和爆炸半径排序风险
3. 为每项风险选择最低但可信的测试层
4. 定义数据、身份、环境与 oracle
5. 设置 gate、回归范围和回滚信号

## 技能包组成

- references/risk-model.md：风险评分和测试层选择
- assets/test-strategy-template.md：测试策略模板
- eval/acceptance.md：风险追溯与发布验收

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据
- INVOCATION.md：标准调用顺序与失败处理
- BASE_PROMPT.md：跨平台执行原则
- agents/openai.yaml：支持平台的界面元数据，如该文件存在
- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 test-strategy-designer/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 高风险项都有测试和可信 oracle
- 环境与数据条件明确
- gate 有责任人、证据和停止条件
- 预发无法覆盖的风险有生产验证和回滚信号

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
