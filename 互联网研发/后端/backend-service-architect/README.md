# 后端服务架构师

> skill id：backend-service-architect
> 当前版本：1.0.0
> 产品状态：可用
> 所属分类：互联网研发 / 后端 / 服务架构

## 产品定位

从业务不变量、数据所有权和故障边界出发，选择最小可演进架构。

## 解决的问题

- 服务边界按技术层切割导致高耦合
- 数据 ownership 不清造成跨库写和一致性事故
- 设计只覆盖正常路径，故障时无法恢复

## 适用场景

- 新业务能力后端设计
- 单体模块化或服务拆分评审
- 高可靠异步工作流和补偿方案

## 不适用范围与边界

- 不得因为流行趋势默认微服务或事件驱动
- 不得让多个服务直接共享写入同一数据而不定义 owner
- 不得只画正常流程，必须设计失败和恢复
- 不得把授权、幂等、可观测性和迁移留到实现后补

## 输入

- 产品需求和现有系统
- 数据、流量、SLO、安全和团队约束
- 可选 API、数据库、部署和事故证据

## 输出

- 后端架构设计
- 边界、数据 ownership 与交互图
- 故障、安全和可观测性方案
- 演进实施与回滚计划

## 工作原理

1. 读取现有仓库、运行拓扑、业务需求、SLO、数据规模、团队边界和部署约束
2. 定义核心领域对象、不变量、命令、查询、状态机和审计需求
3. 划分模块或服务边界及数据 owner，先证明需要分布式边界再引入网络
4. 设计同步 API 与异步事件，说明一致性、排序、重复、重试、超时和补偿
5. 定义认证、对象/字段授权、租户隔离、密钥和敏感数据边界
6. 设计事务范围、幂等、并发控制、缓存、批处理和资源限制
7. 为每个依赖建立失败模式、降级、熔断、队列积压、恢复和人工操作路径
8. 定义日志、指标、trace、审计、健康检查、SLO 和告警
9. 设计部署、配置、迁移、兼容、灰度、回滚和容量计划
10. 输出决策记录、替代方案、权衡、实施切片和验证计划

## 技能包组成

- eval/acceptance.md：可判定验收。
- README.md：中文产品文档与版本记录。

## 使用方式

1. 提供目标、现有代码或生产证据与约束。
2. 让 AI 先读 SKILL.md 和指定 reference。
3. 按仓库实际命令验证，使用 asset 完成交付。

## 验收标准

- 领域不变量和数据 owner 是否明确
- 同步/异步选择是否有理由与失败语义
- 授权、幂等、事务和并发是否完整
- 部署、迁移、观测和回滚是否可执行
