# API 契约设计师

> skill id：api-contract-designer
> 当前版本：1.0.2
> 产品状态：可用
> 所属分类：互联网研发 / 后端 / API 设计

## 产品定位

从消费者行为和领域不变量出发，设计或审查稳定、可演进、可测试的 HTTP 或事件 API 契约。

## 解决的问题

- 接口只定义路径和字段，没有授权、失败、重试和副作用语义
- 空值、默认值、分页、幂等和并发行为不清楚
- 清理字段或类型时无意中制造破坏性变更

## 适用场景

- 从产品需求设计 HTTP API
- 审查 OpenAPI 契约的一致性和兼容性
- 设计事件 Schema、投递、重放和去重语义

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不把内部数据库和服务拓扑泄露成公共契约
- 不承诺无法实现的 exactly-once
- 不以重构名义静默引入 breaking change

## 输入

- 能力需求或现有接口契约
- 消费者、权限模型、仓库规范和兼容约束

## 输出

- API 契约与操作表
- Schema 或 OpenAPI/AsyncAPI 骨架
- 错误目录、示例和兼容发布计划

## 工作原理

1. 识别消费者、信任边界和领域不变量
2. 区分资源、命令、查询和事件
3. 定义授权、Schema、验证、失败与副作用
4. 补齐分页、幂等、并发、重试和可观测性
5. 检查 breaking change 并设计迁移顺序

## 深度执行标准

- 覆盖完整业务理解、主场景与相邻变体、上下游、方案取舍、工具失败、异常、交付、验收和复盘。
- 复杂任务先读 `references/scenario-playbook.md`；存在决策或审批时使用 `assets/decision-record-template.md`。
- 不以篇幅代替深度，每项判断必须关联真实对象、证据、责任和结果。

## 技能包组成

- references/contract-checklist.md：HTTP、事件和破坏性变化检查
- assets/api-review-template.md：契约评审模板
- eval/acceptance.md：语义与兼容验收

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据
- INVOCATION.md：标准调用顺序与失败处理
- BASE_PROMPT.md：跨平台执行原则
- agents/openai.yaml：支持平台的界面元数据，如该文件存在
- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 api-contract-designer/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 每个操作包含成功和失败语义
- 空值、缺省和默认行为明确
- 幂等、并发与重试可测试
- 兼容风险和迁移顺序完整

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
