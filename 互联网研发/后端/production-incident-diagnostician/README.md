# 生产故障诊断师

> skill id：production-incident-diagnostician
> 当前版本：1.0.0
> 产品状态：可用
> 所属分类：互联网研发 / 后端 / 生产故障

## 产品定位

先止损并保全证据，再用可证伪假设缩小范围；恢复服务不等于找到根因。

## 解决的问题

- 团队同时尝试多个变更导致无法判断效果
- 看到最近部署就直接归因
- 服务恢复后遗漏数据不一致和检测缺口

## 适用场景

- 错误率、延迟或服务不可用
- 队列积压、任务失败和依赖超时
- 数据错误、部分用户异常和发布回归

## 不适用范围与边界

- 不得在事故中执行无恢复点的大范围破坏性命令
- 不得把时间相关性直接写成根因
- 不得泄露凭据、个人信息或敏感日志
- 没有授权时只提供诊断与变更建议，不擅自扩大生产操作

## 输入

- 症状、时间、影响和环境
- 日志、指标、trace、部署、配置和依赖状态
- 权限、SLO、runbook 和最近变更

## 输出

- 事件状态与影响
- 时间线和假设记分卡
- 止损/恢复动作与验证
- 根因、数据修复和后续改进

## 工作原理

1. 确认事件指挥、授权、严重度、用户影响、开始时间、范围、数据或安全风险和当前状态
2. 冻结非必要变更并保存日志、指标、trace、部署、配置、feature flag 和依赖证据
3. 选择最安全的止损动作：限流、降级、隔离、暂停消费者、切流、扩容、回滚或关闭功能
4. 建立统一时间线，校准时区、主机时钟和相关变更
5. 列出多个假设，为每个定义预期信号、反证、最低风险检查和状态
6. 沿请求、任务、事件和数据路径关联日志、指标和 trace，避免只搜单条异常
7. 区分触发因素、放大因素、潜在缺陷和观测缺口
8. 执行一个可回滚动作后验证用户指标、错误、延迟、积压、数据和依赖恢复
9. 持续沟通已知、未知、影响、动作、风险和下一更新时间
10. 恢复后完成数据修复、补偿、根因证据、预防和检测改进，并跟踪 owner

## 技能包组成

- SKILL.md：AI 执行入口。
- references/incident-checklist.md：专业检查表。
- assets/incident-record-template.md：交付模板。
- eval/acceptance.md：可判定验收。
- README.md：中文产品文档与版本记录。

## 使用方式

1. 提供目标、现有代码或生产证据与约束。
2. 让 AI 先读 SKILL.md 和指定 reference。
3. 按仓库实际命令验证，使用 asset 完成交付。

## 验收标准

- 影响、严重度、授权和事件角色是否明确
- 证据是否在变更前保存并统一时间线
- 假设是否有预期信号和反证
- 恢复是否从用户、系统和数据三个层面验证
