# Git 安全工作流管理器

> skill id：git-workflow-manager
> 当前版本：2.0.1
> 产品状态：可用
> 所属分类：互联网研发 / Git / 安全工作流

## 产品定位

基于仓库真实状态，安全规划或执行分支、提交、合并、变基、发布流转、冲突处理与恢复。

## 解决的问题

- 按通用分支名猜测仓库流程，容易操作错误目标
- 自动化可能暂存、覆盖或提交用户无关改动
- 危险命令和共享历史重写缺少授权与恢复点

## 适用场景

- 为当前改动选择或创建正确分支
- 准备提交、合并、变基、cherry-pick 或发布流转
- 处理分支分叉、冲突、回退和恢复方案

## 合规与权限

- 合规配置：`product-engineering`（产研合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 未经明确授权不丢弃改动、不 force push、不重写共享历史
- 不暂存或提交无关用户文件
- 不假设 main、master、test、pre 或远端一定存在
- 贵司已存在的 master、main、test、pre 按保护或环境分支处理，不直接开发或提交
- test 不得合并到任何非 test 分支；环境验证后自动返回原开发分支

## 输入

- 仓库路径与目标结果
- 可选分支规范、目标分支、提交范围和发布背景

## 输出

- 仓库状态与风险说明
- 精确 Git 操作计划或已执行结果
- 测试、恢复点和剩余工作

## 工作原理

1. 检查仓库说明、状态、分支、远端、上游、工作树和历史
2. 隔离用户改动与本任务范围
3. 验证真实分支策略和提交范围
4. 选择 merge、rebase、cherry-pick 或 PR
5. 处理冲突并运行针对性验证
6. 报告最终 ref、commit、upstream 和剩余改动
7. 合并或验证完成后安全返回操作前记录的开发分支

## 技能包组成

- references/safety-checks.md：变更前安全检查
- assets/git-change-plan.md：复杂操作计划模板

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据

- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 git-workflow-manager/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 所有目标 ref 和提交范围来自真实仓库
- 用户无关改动保持不变
- 高风险操作有明确授权和恢复点
- 最终状态、测试和剩余工作清楚
- 不在 master、main、test、pre 直接开发，不把 test 合入非 test 分支
- 正常完成后当前分支已回到原开发分支；冲突阻塞时不强制清理

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
