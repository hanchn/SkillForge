# 公司上下文

这里是 SkillForge 面向贵司定制的统一事实入口。所有业务 Skill 在仓库内执行时，应先读取本目录，再使用通用方法。

## 已确认事实

- 公司形态：跨境业务团队。
- 需要覆盖的业务职能：渠道运营、精准营销、创拍视觉（创意、拍摄、视觉、广告）、市场采购、仓储库存、客售前后、互联网研发、数据分析、法律、财务出纳、人事招聘。
- 企业系统版图：OMS、IMS、OFS、CMS、WMS、TMS、CRM、PLM。
- Skill 产品原则：一个 Skill 完成一个可验收的完整业务结果；主角色和周报保留；避免动作级、模板级和重复职责拆分。
- README 是中文产品文档，持续记录能力说明和迭代升级。
- Git 分支治理：禁止在 master、main、test、pre 直接开发或提交；禁止将 test 合并到任何非 test 分支；合并或环境验证完成后安全返回原开发分支。机器可读规则见 `git-policy.yaml`。

## 仍需贵司补充

- 公司、品牌和主体名称。
- 主要国家、站点、币种、时区和法律实体。
- 实际经营渠道、仓库、承运商、支付与广告平台。
- 各系统的实际定义、负责人、主数据和 source of truth。
- 商品类目、SKU 结构、供应模式、履约模式和核心经营指标。
- 部门负责人、审批权限、SLA、例外升级和数据访问边界。

未知项必须保留为空或标记 `待确认`，不得根据行业经验擅自补全。

## 使用顺序

1. 读取 `company-profile.yaml` 获取结构化事实。
2. 涉及 Git 时读取 `git-policy.yaml`，先执行保护分支、环境隔离和返回分支门禁。
3. 读取任务涉及的业务目录 README 和能力地图。
4. 选择能完成完整结果的最少 Skill 组合，并指定一个主 Skill。
5. 公司事实与 Skill 通用方法冲突时，以已确认公司事实为准，同时记录风险和需要审批的偏差。
6. 独立发送 Skill 文件夹时，将本次任务所需的公司上下文快照一并提供。

## 维护规则

- 只有经过业务 owner 确认的事实才能从 `待确认` 改为正式值。
- 系统口径、组织职责、市场、渠道或审批规则变化时，先更新本目录，再升级受影响 Skill。
- 不在这里保存密码、密钥、个人敏感信息、银行凭证或生产访问令牌。
