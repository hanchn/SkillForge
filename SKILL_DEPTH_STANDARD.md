# SkillForge 深度标准

## 核心定义

一个 Skill 应完成一类可验收的完整业务结果，而不是只描述一个动作、一个提示词或一张空表。深度来自真实决策、场景变化和异常处理，不来自篇幅。

## 十二项深度要求

1. 业务理解：解释对象、术语、为什么要做、使用者和错误后果。
2. 场景覆盖：覆盖主场景、相邻高频场景和高风险变体。
3. 范围边界：说明负责、不负责、前置条件和停止条件。
4. 上下游：明确输入方、消费者、系统事实源、交接和责任。
5. 证据口径：定义来源、时间、版本、粒度、可信度和未知项。
6. 决策逻辑：比较方案、适用条件、反证、取舍和可逆性。
7. 工具系统：说明工具输入输出、权限、成本、时效、失败和版本。
8. 异常路径：处理缺数据、冲突、低置信度、超时、部分成功和人工升级。
9. 合规权限：执行分类合规基线和贵司制度，不用免责声明代替门禁。
10. 交付深度：产出下游可直接执行、审批或验证的完整材料。
11. 验收复盘：定义 owner、依赖、指标、护栏、止损、回滚和复盘。
12. 贵司适配：优先使用公司上下文、系统口径、角色和真实项目事实。

## 粒度判断

- 能由同一角色、同一事实集和同一验收结果完成的步骤，应保留在一个 Skill 内。
- 只有当责任人、专业判断、系统权限或最终交付明显不同，才拆分 Skill。
- 主角色负责目标、组合、取舍和验收；专项 Skill 负责可独立验收的专业结果。
- Tool、API、脚本和模型不是独立 Skill，除非其使用本身形成完整业务决策与治理闭环。

## 审计等级

- A：覆盖完整业务理解、场景手册、异常、决策记录、合规和验收，可直接用于复杂任务。
- B：专业流程可靠，但场景变体、工具失败、教学说明或复盘仍需增强。
- C：主要是通用步骤和模板，无法稳定接住复杂任务，必须升级或合并。
- D：职责重复、边界错误或交付不可验收，应删除、迁移或重新设计。

## 维护

运行 `python3 work/rebuild_skillforge_indexes.py` 后同步生成 `SKILL_DEPTH_AUDIT.md` 与 `registry/skill-depth-audit.json`。能力、范围、场景、资源或边界实质变化时，版本系统自动升级对应 Skill。
