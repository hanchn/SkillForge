# SkillForge 全库合规治理

## 原则

每个 Skill 都必须把合规、权限和人工升级作为执行门禁，而不是在结尾附一句免责声明。合规基线随 Skill 文件夹独立分发，并根据所属业务分类生成。

## 两层结构

1. 通用与分类基线：每个 Skill 的 `references/compliance-baseline.md`，覆盖最小必要、权限、隐私、证据、审批、审计和该业务领域的核心风险。
2. 专项规则：高风险 Skill 额外加载专用规则、白名单、Schema 和验收，例如智能打标的标签库、敏感属性拦截和人工复核。

## 决策状态

- `ALLOW`：证据、权限和规则满足，可以在授权范围内继续。
- `REVIEW_REQUIRED`：存在受限事项、证据缺口或高风险，需要指定角色人工审批，Skill 必须暂停相关动作。
- `COMPLIANCE_REJECTED`：任务目的或操作违反禁止规则，不得通过换提示词、拆任务或换工具绕过。

## 分类基线

| 分类 | 重点 |
|---|---|
| 互联网研发 | 数据与隐私、安全、密钥、开源许可、生产变更、AI 评估 |
| 渠道运营 | 商品准入、广告与价格真实性、消费者保护、平台规则 |
| 客售前后 | 商品订单事实、承诺与售后权限、个人信息、敏感咨询、投诉退款和人工升级 |
| 精准营销 | 声明证据、营销同意、退订频控、受众歧视、归因真实性 |
| 创拍视觉 | 广告真实性、肖像版权、商标音乐字体、AIGC 血缘、商品真实性、未成年人 |
| 市场采购 | 主体尽调、制裁出口管制、劳工环境、反贿赂、产品安全 |
| 仓储库存 | 账实审计、职责分离、消防设备、危险品、劳动安全 |
| 数据看板 | 数据来源、最小化、重识别、歧视性分析、口径和不确定性 |
| 法律政务 | 司法辖区、非律师边界、时效证据、保密特权、律师升级 |
| 财务出纳 | 职责分离、支付授权、税务外汇、反欺诈、凭证与密钥 |
| 人事招聘 | 反歧视、候选人员工隐私、背景核验、劳动用工和解雇 |

## 贵司工程治理示例

互联网研发类 Skill 还必须执行 `公司上下文/git-policy.yaml`：

- 禁止在 `master`、`main`、`test`、`pre` 直接开发或提交。
- 禁止将 `test` 合并到任何非 `test` 分支。
- 合并或环境验证前记录原开发分支，完成后安全返回该分支。
- 冲突或未保存工作阻止切换时保留现场并报告，不得通过破坏性清理强行返回。

这类规则属于公司流程、权限和工程控制合规，与法律、平台、数据、内容合规使用同一门禁机制。后续可继续在 `公司上下文/` 增加发布、数据库、财务审批、客户数据、广告投放等机器可读制度。

## 自动维护

运行 `python3 work/rebuild_skillforge_indexes.py` 时会先执行 `sync_skill_compliance.py`：

- 为所有正式 Skill 生成或刷新分类合规基线。
- 在 `SKILL.md` 注入强制读取要求。
- 在中文 README 展示合规配置和能力边界。
- 在 `skill.json` 写入机器可读合规配置。
- 将合规文件加入 ZIP 打包清单。
- 合规内容实质变化后，由版本系统自动升级 Skill 版本。

## 边界

Skill 是规则和判断层，不能单独保证真实权限隔离、数据不外泄、授权真伪、不可篡改日志或最终法律结论。必须与贵司系统控制、审批、法务、审计和安全机制组合使用。
