# 业务指标诊断师

> skill id：metric-diagnostics
> 当前版本：1.0.4
> 产品状态：可用
> 所属分类：数据 / 业务分析 / 指标诊断

## 产品定位

先验证指标口径和数据质量，再量化拆解异常变化，并区分已确认驱动因素与待验证假设。

## 解决的问题

- 指标还没验证就开始讲原因故事
- 不完整周期、平均比率、小基数和分母漂移造成错误结论
- 只看分组百分比变化，不计算对总体的真实贡献

## 适用场景

- 诊断指标上涨、下跌、异常或预测未达
- 解释漏斗、收入、活跃和留存变化
- 核对两个仪表盘或查询结果为什么不同

## 合规与权限

- 合规配置：`data-analytics`（数据分析合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不把相关性写成因果
- 不在未说明情况下比较完整与不完整周期
- 不使用无权重的小样本变化代表总体驱动

## 输入

- 指标定义与比较周期
- 可选查询、CSV、仪表盘、变更记录和分群维度

## 输出

- 指标诊断报告
- 贡献度分解表
- 假设证据记分卡
- 决策含义和下一证据

## 工作原理

1. 锁定人群、粒度、时区、过滤和归因窗口
2. 复算并检查新鲜度、重复、连接、迟到与分母
3. 量化绝对、相对及分子分母变化
4. 先列竞争假设再按指标代数和分群拆解
5. 验证时间、暴露和机制
6. 将结论分为确认、贡献、排除和未决

## 深度执行标准

- 覆盖完整业务理解、主场景与相邻变体、上下游、方案取舍、工具失败、异常、交付、验收和复盘。
- 复杂任务先读 `references/scenario-playbook.md`；存在决策或审批时使用 `assets/decision-record-template.md`。
- 不以篇幅代替深度，每项判断必须关联真实对象、证据、责任和结果。

## 技能包组成

- references/diagnostic-method.md：校验、分解和证据分类
- assets/diagnostic-report-template.md：诊断报告模板
- eval/acceptance.md：口径、贡献和因果边界验收

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据
- INVOCATION.md：标准调用顺序与失败处理
- BASE_PROMPT.md：跨平台执行原则
- agents/openai.yaml：支持平台的界面元数据，如该文件存在
- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 metric-diagnostics/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 口径和数据质量先于归因
- 总体变化能与分解结果对账
- 贡献度处理权重和小基数
- 观察与推断清楚分开

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
