# SkillForge 项目治理

> skill id：skillforge-project-governance
> 当前版本：2.0.2
> 产品状态：持续维护
> 所属分类：项目根目录 / 项目治理

## 产品定位

SkillForge 仓库的项目级治理产品。它定义业务 skill 如何分类、创建、验收、注册、升级和独立分发。

## 解决的问题

- 不同 AI 接手仓库后对目录、文件职责和质量标准理解不一致
- 新增 skill 容易退化为只有几段泛化提示词的空包
- skill 更新后注册表、模板、产品文档和验收规则不同步

## 适用场景

- 新增、重构或升级任何 SkillForge skill
- 决定一个新能力应放在哪个业务分类
- 审查 skill 是否具备真实工作流、证据边界和可判定验收
- 准备把某个 skill 文件夹单独发给其他人或平台

## 合规与权限

- 合规配置：`repository-governance`（项目治理合规基线）。
- 执行前必须读取 `references/compliance-baseline.md`，完成司法辖区、数据、权限、授权、人工复核和审计检查。
- Skill 只能提供第一层规则约束，不能替代系统权限、真实授权、正式审批或专业法律意见。

## 不适用范围与边界

- 不替代具体业务 skill 的专业工作流
- 不把业务 skill 放入平台私有目录
- 不允许仅更新 SKILL.md 而忽略 README、skill.json、验收和索引的一致性

## 输入

- 目标 skill 路径或业务领域
- 新增、升级、迁移或分发需求
- 现有项目规则与技能包证据

## 输出

- 分类和目录决策
- 技能包结构与质量要求
- 需要同步更新的文件清单
- 验证与分发结论

## 工作原理

1. 先读取项目索引、注册表和当前治理规则
2. 判断变更属于项目规则、模板还是具体业务 skill
3. 检查触发元数据、专业工作流、资源、边界、验收和版本记录
4. 同步注册表、产品文档、模板与分类导航
5. 运行结构、schema、路径与脚本验证

## 深度执行标准

- 覆盖完整业务理解、主场景与相邻变体、上下游、方案取舍、工具失败、异常、交付、验收和复盘。
- 复杂任务先读 `references/scenario-playbook.md`；存在决策或审批时使用 `assets/decision-record-template.md`。
- 不以篇幅代替深度，每项判断必须关联真实对象、证据、责任和结果。

## 技能包组成

- SKILL.md：AI 治理入口
- skill.json：机器可读产品元数据
- README.md：中文产品文档与版本记录
- ARCHITECTURE.md：仓库架构
- assets/project-rules.md：强制规则
- eval/acceptance.md：治理验收标准

通用入口文件：

- SKILL.md：AI 执行入口与专业工作流
- skill.json：机器可读元数据
- INVOCATION.md：标准调用顺序与失败处理
- BASE_PROMPT.md：跨平台执行原则
- agents/openai.yaml：支持平台的界面元数据，如该文件存在
- README.md：当前中文产品文档与版本记录

## 使用方式

1. 将整个 skillforge-project-governance/ 目录交给支持 skill 的 AI 或复制到对应技能目录。
2. 让 AI 先读取 SKILL.md；README.md 用于人类理解产品能力和升级历史。
3. 按任务需要提供输入证据，执行后使用 eval/acceptance.md 验收。
4. 若修改能力、边界、输入输出或资源，应同步更新 README.md 的当前说明和版本记录。

## 验收标准

- 规则能指导另一个 AI 完成同类变更
- 注册表、模板、产品文档和技能包不存在相互冲突
- 新规则明确指出作用范围和升级路径

## 版本与升级规则

- 当前能力以本 README 和 skill.json 的版本为准。
- 每次能力、工作流、资源、输入输出或边界发生实质变化，必须提升版本并补充下方记录。
- 仅修正文案错字或链接时可不提升版本，但仍应保证产品说明与实际文件一致。
- 历史记录只写已发生变化，不把计划中的能力写成已经交付。
