# Skill 组合治理与合并标准

## 目标

SkillForge 不以 Skill 数量为目标，而以“能否稳定代替一个真实工作结果”为目标。大而全不等于拆得多，而是完整覆盖公司业务，同时让每个 Skill 有清楚的责任边界。

## 新建判断

满足以下条件才新建 Skill：

1. 有独立且高频的触发场景。
2. 有区别明显的专业方法或知识依据。
3. 有可独立验收的交付结果。
4. 有独立风险、权限或合规边界。
5. 无法通过升级现有 Skill 自然完成。

任一动作只改变同一流程的阶段、语言、消息格式或单个模板时，优先作为现有 Skill 的步骤、资源或输出变体。

## 保留拆分的典型情况

- OMS、IMS、OFS、CMS、WMS、TMS、CRM、PLM：核心对象、状态机和系统边界不同。
- 美国、欧盟、进出口法律合规：司法辖区、主管机关和高风险依据不同。
- React 与 Vue3：框架运行模型、工程方法和诊断依据不同。
- 主角色与周报：一个负责持续经营决策，一个负责固定周期证据化汇报，触发与交付不同。

## 应合并的典型情况

- 同一坐席完成的咨询、推荐、多语言回复、FAQ 回写和线索交接。
- 同一库存计划流程中的监测、告警、测算、补货建议和处置复盘。
- 仅有“写通知、做签到、收反馈”等单一步骤，而没有完整业务结果的能力。

## 公司定制要求

- 每次执行先读取 `公司上下文/`，不得把行业默认值当成贵司事实。
- 输出使用贵司部门、系统、状态、指标和审批语言。
- 每个 Skill 明确与上下游部门及 OMS、IMS、OFS、CMS、WMS、TMS、CRM、PLM 的接口。
- 对未知 owner、系统口径、渠道、市场、权限和 SLA 明确标记 `待确认`。
