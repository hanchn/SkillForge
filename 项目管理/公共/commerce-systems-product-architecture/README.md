# 跨境业务系统总产品架构师

> skill id：commerce-systems-product-architecture  
> 当前版本：1.0.0  
> 产品状态：可用  
> 所属分类：项目管理 / 公共

## 产品定位

设计 OMS、IMS、OFS、CMS、WMS、TMS、CRM、PLM 等系统的职责地图、主数据归属和端到端协同。

## 解决的问题

- 工作依赖个人经验，输入、过程和交付口径不稳定。
- 只关注局部动作，缺少边界、上下游、异常、数据和经营结果闭环。
- 建议无法追溯证据，也没有负责人、验收、止损或演进机制。

## 适用场景

- 跨境业务系统蓝图
- 系统边界重构
- 新系统立项和集成治理

## 不适用范围与边界

- 不得先按现有部门或系统名称切边界，必须先锁定业务能力与贵司缩写口径。
- 关键口径或权限缺失时，本 Skill 可以完成诊断与方案，但不能虚构执行结果。

## 输入

- 必需：业务目标、范围、对象、现状证据和已知约束。
- 建议：历史数据、流程或系统资料、目标值、owner、预算、SLA、合规要求。
- 可选：竞品、用户反馈、事故、实验、财务和上下游契约。

## 输出

- 业务能力与系统地图
- 主数据和状态 ownership 矩阵
- 跨系统流程、契约与演进路线
- 决策记录、执行计划、指标验收、风险与未知项。

## 工作原理

1. 确认贵司缩写、业务范围、用户角色、当前系统、目标和明确不在范围内的事项
2. 从端到端业务旅程提炼能力地图，不按现有菜单、组织或数据库表直接切系统
3. 定义核心对象、标识、主数据 source of truth、状态机、命令、事件和审计要求
4. 划清本系统负责、不负责及与上下游重叠区域，建立 RACI 和数据 ownership 矩阵
5. 设计正常、异常、补偿、人工介入、对账、重放和幂等路径及跨系统 SLA
6. 按 MVP、增强、平台化阶段输出产品路线、依赖、验收、迁移和治理机制

## 专业检查维度

- 业务能力和端到端旅程
- 系统职责与不负责范围
- 主数据和 source of truth
- 命令、事件、状态和异常
- 集成、SLA、审计和分阶段建设

## 技能包组成

- `SKILL.md`：AI 执行入口与强制工作流。
- `references/professional-checklist.md`：专业检查表和失败模式。
- `assets/delivery-template.md`：可直接复用的交付模板。
- `examples/README.md`：调用示例。
- `eval/acceptance.md`：可判定验收标准。
- `README.md`：中文产品文档与迭代记录。

## 使用方式

1. 提供目标、范围、现状证据和限制条件。
2. 要求 AI 先读 `SKILL.md` 与 reference，再检查真实资料。
3. 让 AI 使用 asset 交付，并按 eval 自检；涉及写入或上线时另行确认权限。

## 验收标准

- 范围、术语、时间和数据口径明确。
- 关键结论均有证据、公式或清晰推理支撑。
- 方案覆盖正常路径、异常路径、上下游和责任边界。
- 动作有优先级、owner、依赖、指标、护栏和停止条件。
- 未知项、不确定性、风险和恢复方案被明确记录。
